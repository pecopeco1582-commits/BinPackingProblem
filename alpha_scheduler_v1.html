What this does

You have a pile of computer jobs to run and a set of servers to run them on. The tool fits the jobs onto as few servers as possible — then solves the same puzzle two ways so you can compare them.

The goal: every job gets a server, no server is overloaded, and you use as few servers as you can — because idle servers still cost money.

A It's like packing suitcases

Each job takes up some space (a bit of processing power, some memory, some storage). Each server is a suitcase with a fixed size. The trick is fitting everything into the fewest suitcases. Easy with a few items — surprisingly hard once there are many.

B Two ways to solve it

A normal computer solves the packing puzzle the usual way — quickly. A quantum method (its real name is QAOA) solves the very same puzzle using a simulated quantum computer. The interesting part is watching how the two compare on speed.

C What needs to be running

This page talks to a small program on your own computer (at localhost:8000). Start that program first, then come back and click the compare button. If it isn't running, you'll see a friendly error.

Step 2 of 5

The jobs

There are four kinds of job. Each one needs a certain amount of a server's resources and runs for a certain length of time. You choose how many of each to schedule.

web_api
1 core · 1 GB · 5 sec
A quick web request. Tiny and fast — you usually have lots of these.
batch
2 cores · 2 GB · 5 min
A long number-crunching job. The heaviest kind — one of these can nearly fill a server on its own.
memory
1 core · 3 GB · 1 min
A job that needs lots of memory. Uses most of a server's memory, so it doesn't share well.
io_heavy
1 core · 1 GB · 3 min
A job that reads and writes lots of data. Hungry for storage and network.
Tip: Start with 3 web_api + 1 batch on 3 servers. That's the default, and it's small enough for the quantum method to finish quickly.

Step 3 of 5

Servers & size

The slider sets how many servers are available. There's one thing to keep in mind: the bigger the problem, the slower the quantum method gets — fast.

A What one server holds

Every server is the same size:

One server
2 cores  ·  4 GB memory  ·  40 GB storage  ·  10 Gbps network

No server is ever allowed to go over any of these limits. The tool finds a way to fit everything within them, using as few servers as possible.

B Why size matters for the quantum side

The puzzle is made of small yes/no decisions — one for each job-and-server pairing:

Problem size
decisions = (jobs × servers) + servers

The normal computer barely notices. But the quantum method needs one "qubit" per decision, and every extra qubit doubles the possibilities it has to sift through. So keep the numbers small — the live counter on the left turns orange when you're pushing it.

Running it

The exact steps from nothing to a result.

1 Start the helper program

Open a terminal in the project folder and run:

# Go to the project folder
cd "capstone final"

# Install what's needed (first time only)
pip install fastapi uvicorn scipy qiskit-optimization

# Start it
python -m uvicorn alpha2:app --reload

When you see Uvicorn running on http://0.0.0.0:8000, leave that window open.

2 Open this page

Open scheduler_v9.html in Chrome or Firefox. The page itself needs no server — it just talks to the helper program over localhost:8000.

3 Choose your jobs

On the left, type how many of each job you want (at least one). Watch the Problem size box near the bottom — keep it in the green.

4 Choose how many servers

Drag the servers slider. A little extra room often lets the tool find a tidier packing.

5 Click “Compare the two methods”

The tool runs both solvers on your setup. The quantum one can take a few seconds. The results appear on the right.

Step 5 of 5

Reading the results

You get a side-by-side table, a short explanation of why the quantum method slows down, and a one-line takeaway.

A Servers used

How many servers each method needed. Fewer is better — it means the jobs were packed more tightly. Often both methods tie on small problems.

B Total time to finish

How long until every job is done, once they're running. A single batch job (5 minutes) usually sets this.

C Time to find the answer

How long each method itself took to think. This is the headline: the normal computer answers in a flash, while the quantum method takes much longer — and the gap grows fast as the problem gets bigger.

D Why the quantum method slows down

A panel shows the problem size turning into qubits, and qubits turning into a huge number of possibilities. That number is the whole reason the quantum method is slow here.

About this project

Packing Jobs onto Servers

A small teaching tool. It takes a set of computer jobs and fits them onto the fewest servers possible — making sure no server runs out of processing power, memory, storage, or network along the way.

The point of the tool is the comparison. It solves the exact same packing puzzle two ways: with an ordinary solver running on a normal computer, and with a quantum method called QAOA running on a simulated quantum computer. Then it shows you, side by side, which used fewer servers and — the interesting part — how much longer the quantum method took.

The honest lesson: on small problems the normal computer wins easily. The quantum approach only starts to matter on problems far too big to run on a laptop. This tool lets you watch that gap appear for yourself.

What's under the hood
The helper program
A small web service on your computer. It takes your jobs and server count, runs both methods, and sends back the comparison this page shows.
The normal method
A fast, well-known packing recipe (best-fit). It places each job on the server where it fits most snugly. Quick even on big problems.
The quantum method (QAOA)
Solves the same puzzle on a simulated quantum computer. Accurate on small problems, but it slows down sharply as the problem grows — which is exactly what the comparison shows.
This page
A single file, no installation. It just talks to the helper program and lays the two results next to each other.
